📦 SECTION 1 — OOP & Java Fundamentals

Q1. What is the difference between == and .equals()?

A: == compares references (memory addresses). .equals() compares logical content if overridden. 
String a = new String("hello");
String b = new String("hello");
a == b;       // false — different objects
a.equals(b);  // true  — same content
Trap: String literals are interned, so "hello" == "hello" is true. Never rely on this for comparison.

Q2. What is the contract between equals() and hashCode()?

A: If a.equals(b) is true, then a.hashCode() == b.hashCode() must also be true.

The reverse is not required (hash collision is allowed).

Why it matters: HashMap uses hashCode() to find the bucket, then equals() to find the key. If you override equals() without hashCode(), two "equal" objects land in different buckets — HashMap.get() silently returns null.

Q4. Explain final, finally, finalize.

  • finally: block that always runs after try/catch, even on exception
  • finalize(): deprecated Object method called by GC before collection — never rely on it

Q5. What is autoboxing and what are its pitfalls?

A: Automatic conversion between primitives and wrapper types (int ↔ Integer). 
Integer a = 127; Integer b = 127; a == b; // true  — cached range [-128, 127]
Integer c = 128; Integer d = 128; c == d; // false — new objects outside cache
Pitfall 1: == comparison on Integer outside cache range is wrong.
Pitfall 2: Unboxing null throws NullPointerException.
Pitfall 3: Performance — tight loops with boxed types create excessive GC pressure.

Q7. How does HashMap work internally?

A:
  1. hashCode() is computed for the key, then spread with (h ^ (h >>> 16)) to reduce collisions.
  2. Bucket index = hash & (capacity - 1) (capacity is always power of 2).
  3. Each bucket is a linked list of Entry nodes.
  4. On collision, new entry is chained. On get, the bucket is traversed with equals().
  5. Java 8+: When a bucket chain exceeds 8 nodes, it converts to a Red-Black Tree → O(log n) lookup instead of O(n).
  6. Load factor (default 0.75): when size > capacity * 0.75, the map rehashes into a new array double the size.

Q11. What is Optional and when should you NOT use it?

A: Optional<T> is a container that may or may not contain a value — forces callers to handle the absent case. 
Optional.ofNullable(map.get(key))
    .map(String::toUpperCase)
    .orElse("default");
Do NOT use Optional:
  • As method parameters — use @Nullable annotation instead
  • As field types — wastes memory, breaks serialization
  • In collections — List<Optional<T>> is an antipattern

Q12. What are Records (Java 14+) and Sealed Classes (Java 17)?

sealed interface Shape permits Circle, Rectangle, Triangle {}

Used with pattern matching switch — compiler enforces exhaustive handling of all permitted types.

🔒 SECTION 3 — Concurrency & Multithreading

Q14. What is the difference between Runnable, Callable, and Future?

  • Runnable: no return value, no checked exception
  • Callable<V>: returns V, can throw checked exception
  • Future<V>: handle to an async task — get() blocks until done, isDone() checks status

Q15. What is the Java Memory Model and happens-before?

A: The JMM defines when a write by Thread A is guaranteed visible to Thread B. Without synchronisation, the JVM and CPU can reorder instructions and cache values. Happens-before rules:
  • Unlock → subsequent lock of same monitor
  • volatile write → subsequent read of same variable
  • Thread.start() → any action in the new thread
  • Thread.join() → any action after the join returns
If no happens-before relationship exists between a write and a read, the read may see any value — including stale data.

Q16. When is volatile sufficient vs when do you need synchronized?

  • volatile: guarantees visibility. Use for simple flags or single-variable state.
  • synchronized: guarantees visibility + atomicity of compound operations.
// volatile WRONG for compound operation:
volatile int count;
count++; // read-modify-write — NOT atomic, race condition!

// Correct:
AtomicInteger count = new AtomicInteger();
count.incrementAndGet(); // CAS — atomic

Q17. What is synchronized? What are the different forms?

// Method lock — locks on 'this'
public synchronized void method() { }

// Block lock — lock on specific object (preferred, finer granularity)
private final Object lock = new Object();
public void method() {
    synchronized (lock) { }
}

// Static method — locks on Class object
public static synchronized void staticMethod() { }

Q19. What is a deadlock? How do you prevent it?

A: Deadlock: Thread A holds Lock 1 and waits for Lock 2. Thread B holds Lock 2 and waits for Lock 1. Prevention strategies:
  1. Lock ordering — always acquire locks in the same global order
  2. tryLock with timeout — if lock not acquired in N ms, back off and retry
  3. Lock-free data structures — AtomicInteger, ConcurrentHashMap
  4. Single lock — if possible, use one lock instead of multiple

Q21. What is CompletableFuture? Show a real pattern.

// Chain async operations
CompletableFuture<String> result = CompletableFuture
    .supplyAsync(() -> fetchUser(userId))         // runs on ForkJoinPool
    .thenApply(user -> user.getName())             // transform result
    .thenCompose(name -> fetchOrders(name))        // flat-map async
    .exceptionally(ex -> "default");               // error handler

CompletableFuture<String> userFuture   = CompletableFuture.supplyAsync(() -> fetchUser(id));

CompletableFuture.allOf(userFuture, orderFuture).join();

Q22. What is ThreadLocal? When is it useful and when is it dangerous?

A: ThreadLocal<T> gives each thread its own isolated copy of a variable. Useful: Storing per-request context (e.g., authenticated user, correlation ID, database connection). Dangerous:
  • Thread pool reuse — a thread's ThreadLocal value from Request 1 can leak to Request 2 if not cleared.
  • Memory leaks — if the ThreadLocal reference is in a static field and threads are long-lived.
// Always clean up in a try/finally
try {
    REQUEST_CONTEXT.set(ctx);
    // handle request
} finally {
    REQUEST_CONTEXT.remove(); // critical!
}

🏗️ SECTION 4 — JVM Internals & GC

Q23. Describe the JVM memory structure.

  • Heap: Objects live here. Divided into Young Generation (Eden + Survivor) and Old Generation (Tenured).
  • Stack: Per-thread. Stores stack frames (local variables, operand stack, method references).
  • Metaspace (Java 8+): Class metadata, method bytecode. Replaces PermGen. Grows dynamically.
  • Code Cache: JIT-compiled native code.
  • PC Register: Per-thread program counter.

Q24. How does Garbage Collection work? What is G1GC?

A: GC reclaims unreachable objects. Key phases:
  1. Mark — trace from GC roots (stack, static fields) to find live objects
  2. Sweep — reclaim memory of unmarked objects
  3. Compact — defragment heap (some collectors)
G1GC (default Java 9+):
  • Divides heap into fixed-size regions (not fixed-size generations)
  • Collects regions with most garbage first — hence "Garbage First"
  • Targets a configurable pause goal (-XX:MaxGCPauseMillis=200)
  • Good for heaps 4GB–32GB
ZGC / Shenandoah: Sub-millisecond pauses — use for latency-critical services.

Q26. What is a memory leak in Java? Give an example.

A: A memory leak is when objects are no longer needed but are still referenced — GC cannot collect them. Classic examples:
  • Static List or Map that grows indefinitely
  • ThreadLocal not removed on thread reuse
  • Listeners/callbacks registered but never deregistered
  • Cache without eviction policy
// Subtle leak: custom equals/hashCode not overridden
Map<MyKey, Value> cache = new HashMap<>();
cache.put(new MyKey("id1"), value); // stored
cache.get(new MyKey("id1"));        // different object, different hash → never found
// Map grows forever, values never retrieved or removed

Q27. What is Inversion of Control (IoC) and Dependency Injection (DI)?

IoC: The framework controls object creation and lifecycle — not your code. DI: Dependencies are injected by the container, not created inside the class. 
// Without DI — tight coupling, untestable
public class OrderService {
    private final PaymentService payment = new PaymentService(); // bad
}

// With DI — loose coupling, mockable
@Service
public class OrderService {
    private final PaymentService payment;
    public OrderService(PaymentService payment) { // injected
        this.payment = payment;
    }
}

Q33. What is @Autowired and how does Spring resolve ambiguity?

Spring first matches by type. If multiple beans of the same type exist:

  2. Use @Qualifier("beanName") on the injection point
  3. Name the parameter to match the bean name (constructor injection)

Q34. What is Spring AOP? What is a Pointcut, Advice, and Aspect?

  • Aspect: module that cuts across concerns (logging, transactions, security)
  • Pointcut: expression that selects which methods to intercept
  • Advice: code that runs at the join point
@Aspect @Component
public class LoggingAspect {

    @Around("execution(* com.example.service.*.*(..))")
    public Object log(ProceedingJoinPoint pjp) throws Throwable {
        log.info("Calling {}", pjp.getSignature());
        Object result = pjp.proceed();
        log.info("Done {}", pjp.getSignature());
        return result;
    }
}
AOP works via JDK dynamic proxies (interface-based) or CGLIB proxies (class-based). Self-invocation within the same bean bypasses the proxy — AOP will not trigger.

Q39. What is the N+1 problem? How do you fix it?

Q41. Why does @Transactional not work on private methods or self-invocation?

Spring @Transactional works via proxy. When you call this.someMethod() inside the same class, you call the raw object — not the proxy — so the interceptor never runs.

// BROKEN — self-invocation bypasses proxy
public void processOrder() {
    this.saveOrder(); // calls raw bean, @Transactional ignored!
}

@Transactional
public void saveOrder() { ... }

// FIX 2: Extract saveOrder() to a separate @Service bean

Q42. What is the difference between EAGER and LAZY loading?

  • EAGER: Related entity fetched immediately in the same query
  • LAZY: Related entity fetched on first access (proxy)
Default: @ManyToOne = EAGER, @OneToMany = LAZY
Best practice: Always use LAZY. Fetch eagerly only via JOIN FETCH when you know you need the data — prevents unwanted joins.

Q43. What is optimistic vs pessimistic locking in JPA?

Optimistic: No DB lock. Uses a @Version column. If two transactions read and update the same row, the second commit detects the version mismatch and throws OptimisticLockException. 
@Version
private int version; // JPA auto-increments on each update
Pessimistic: DB-level lock (SELECT FOR UPDATE). Blocks concurrent readers. 
@Lock(LockModeType.PESSIMISTIC_WRITE)
Optional<Account> findById(Long id);
Use optimistic for low-contention reads. Use pessimistic when you cannot afford version conflicts (financial transfers).

Q54. What is the Saga pattern and when do you use it?

Problem: Distributed transactions across microservices — you can't use a single ACID transaction. Saga: A sequence of local transactions, each publishing an event. On failure, compensating transactions undo previous steps. Orchestration: A central orchestrator directs each step. Choreography: Each service reacts to events — no central coordinator.
Use Saga when: transferring funds across services, booking travel (hotel + flight + car), order fulfillment (inventory + payment + shipping).

Q57. What are the transaction isolation levels and what anomalies do they prevent?

Q58. What is a database index? How does a B-Tree index work?

A: An index is a separate data structure that maps column values → row locations, allowing the DB to find rows without a full table scan. B-Tree (default in Postgres/MySQL):
  • Balanced tree where leaf nodes contain the indexed values and pointers to heap rows
  • Supports: =, <, >, BETWEEN, LIKE 'prefix%', ORDER BY
  • Does NOT help: LIKE '%suffix', functions on the column (LOWER(name) = 'x')
Multi-column index: CREATE INDEX ON orders(user_id, status) — usable for queries filtering on user_id alone, or user_id + status, but NOT status alone (left-prefix rule).

Q59. What is the difference between a clustered and non-clustered index?

  • Clustered (Primary Key in InnoDB / PGSQL heap): The actual table data is stored in index order. One per table.
  • Non-clustered (Secondary): Separate structure pointing back to the heap. Multiple per table. Extra lookup needed to fetch full row (heap fetch).
In PostgreSQL, all indexes are non-clustered by default. The table heap is separate. Use CLUSTER to physically reorder once, but it's not maintained automatically.

Q60. How do you find and fix a slow query?

-- Step 1: EXPLAIN ANALYZE to see actual query plan
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM orders WHERE user_id = 42;

-- What to look for:
-- Seq Scan → missing index
-- Nested Loop with large rows → consider Hash Join
-- "actual rows" << "estimated rows" → stale statistics (run ANALYZE)
-- "Buffers: shared hit=0 read=50000" → reading from disk, not cache
Fix checklist:
  1. Add index on the filter/join column
  2. Run ANALYZE to refresh planner statistics
  3. Rewrite correlated subquery as JOIN
  4. Add LIMIT to avoid full scans on large result sets
  5. Use a covering index (include all SELECT columns so heap fetch is skipped)

Q61. What is a covering index?

An index that contains all columns needed by the query — the query is answered entirely from the index without touching the table heap.

-- Query: SELECT email FROM users WHERE status = 'ACTIVE'
-- Covering index: includes both the filter column AND the select column
CREATE INDEX idx_users_status_email ON users(status) INCLUDE (email);
-- "Index Only Scan" — zero heap fetches

Q62. What is the difference between WHERE, HAVING, and when to use each?

  • WHERE — filters rows before grouping. Can use indexes.
  • HAVING — filters groups after GROUP BY. Cannot use indexes directly.
-- Correct: filter on non-aggregated column → use WHERE
SELECT user_id, COUNT(*) FROM orders WHERE status = 'PAID' GROUP BY user_id;

-- Correct: filter on aggregate result → use HAVING
SELECT user_id, COUNT(*) FROM orders GROUP BY user_id HAVING COUNT(*) > 10;

Q63. What is a database connection pool and why is it critical?

Creating a DB connection is expensive (~50–100ms). A pool maintains pre-created, reusable connections.

HikariCP (Spring Boot default — fastest pool): 
spring.datasource.hikari:
  maximum-pool-size: 20      # max concurrent connections
  minimum-idle: 5            # keep 5 warm when idle
  connection-timeout: 30000  # fail fast if no connection available in 30s
  idle-timeout: 600000       # evict idle connections after 10 min
  max-lifetime: 1800000      # recycle connections after 30 min (avoids DB-side timeout)
Sizing rule of thumb: pool_size = (core_count * 2) + effective_spindle_count. For a 4-core machine with SSD: ~10 connections. More is not better — excess connections queue and contend.

📨 SECTION 13 — Kafka & Messaging

Q65. What is Kafka's architecture?

  • Topic: Named log. Data is appended-only, immutable.
  • Partition: Topics split into N partitions. Each partition is an ordered, immutable log. Enables parallelism.
  • Offset: Each message in a partition has a monotonically increasing offset. Consumers track their offset.
  • Consumer Group: Group of consumers that together consume all partitions of a topic. Each partition is assigned to exactly one consumer in the group → parallel processing.
  • Broker: Kafka server. A cluster has multiple brokers. Each partition has one leader broker and N-1 replicas.

Q66. How does Kafka guarantee message ordering?

Within a partition: Strictly ordered. Messages are appended and consumed in order. Across partitions: No ordering guarantee. Implication: To guarantee ordering for a logical entity (e.g., all events for orderId=42), use the orderId as the partition key. All messages for the same key land in the same partition. 
producer.send(new ProducerRecord<>("orders", orderId.toString(), payload));
// Same orderId → same partition → strictly ordered

Q67. What is the difference between at-most-once, at-least-once, exactly-once?

Lost messages?Duplicate messages?
At-most-onceYesNo
At-least-onceNoYes (on retry)
Exactly-onceNoNo
Kafka exactly-once requires:
  1. Idempotent producer (enable.idempotence=true) — deduplicates retries at broker
  2. Transactional API — atomic write across multiple partitions/topics
  3. Consumer reads with isolation.level=read_committed — ignores uncommitted messages
In practice, most systems implement at-least-once + idempotent consumer — simpler and nearly as strong.

Q68. What is a Consumer Group and how does rebalancing work?

Each consumer in a group owns a subset of partitions. When a consumer joins/leaves:

  1. Rebalance triggered — all consumers in the group stop consuming briefly
  2. Group Coordinator (a broker) reassigns partitions to remaining consumers
  3. Consumers resume from their last committed offset
Problem: Rebalance causes a pause — "stop the world" moment. Minimize with:
  • session.timeout.ms and max.poll.interval.ms tuning
  • Cooperative (incremental) rebalancing (Kafka 2.4+) — only reassigns partitions that changed, not all

Q69. What is a Dead Letter Queue (DLQ)?

When a consumer fails to process a message after N retries, instead of blocking the partition, the message is published to a DLQ topic. Processing continues on the main topic.

@KafkaListener(topics = "payments")
public void process(PaymentEvent event) {
    try {
        paymentService.process(event);
    } catch (Exception e) {
        kafkaTemplate.send("payments.dlq", event); // send to DLQ
    }
}
DLQ messages are monitored and either reprocessed manually or auto-retried by a DLQ consumer with back-off.

Q72. What is cache stampede (dog-pile effect) and how do you prevent it?

Problem: A popular cache key expires. Hundreds of concurrent requests all miss cache simultaneously and hit the database — spiking load. Solutions:
  1. Probabilistic early expiration: Refresh the key slightly before it expires (background refresh)
  2. Mutex lock: First request to miss acquires a Redis lock, fetches from DB, repopulates. Others wait briefly.
  3. Jitter on TTL: Instead of all keys expiring at exactly the same time, add random jitter: TTL = base + random(0, base * 0.2)

Q78. Singleton — how to implement it correctly in Java?

// Best approach: Enum singleton (thread-safe, serialization-safe)
public enum AppConfig {
    INSTANCE;
    private final String value = loadConfig();
}

// Alternative: Double-checked locking
public class Singleton {
    private static volatile Singleton instance;
    private Singleton() {}
    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) instance = new Singleton();
            }
        }
        return instance;
    }
}
volatile prevents instruction reordering — without it, another thread may see a partially constructed object.
In Spring: every @Bean is a singleton by default — you rarely need to implement this yourself.

Q79. Builder Pattern — when and how?

Use when constructing objects with many optional parameters (avoids telescoping constructors).

public class HttpRequest {
    private final String url;
    private final String method;
    private final Map<String, String> headers;
    private final int timeoutMs;

    private HttpRequest(Builder b) {
        this.url = b.url; this.method = b.method;
        this.headers = b.headers; this.timeoutMs = b.timeoutMs;
    }

    public static class Builder {
        private final String url;
        private String method = "GET";
        private Map<String, String> headers = new HashMap<>();
        private int timeoutMs = 5000;

        public Builder(String url) { this.url = url; }
        public Builder method(String m) { this.method = m; return this; }
        public Builder header(String k, String v) { headers.put(k, v); return this; }
        public Builder timeout(int ms) { this.timeoutMs = ms; return this; }
        public HttpRequest build() { return new HttpRequest(this); }
    }
}
// Usage:
HttpRequest req = new HttpRequest.Builder("https://api.example.com")
    .method("POST").header("Auth", "Bearer xyz").timeout(3000).build();
Lombok's @Builder generates this automatically.

📐 SECTION 17 — System Design Fundamentals (Rapid Fire)

Q84. What is CAP theorem?

In a distributed system, you can only guarantee two of three:

  • Consistency — every read gets the latest write
  • Availability — every request gets a response
  • Partition Tolerance — system works despite network partition
Network partitions are unavoidable → you always sacrifice either C or A during a partition.
- CP systems: Zookeeper, HBase, etcd — prioritise consistency
- AP systems: Cassandra, CouchDB, DynamoDB — prioritise availability

Q85. What is eventual consistency?

After a write, replicas will eventually converge to the same value — but may return stale data immediately after the write. Acceptable when:

  • Reading user profiles, social feeds, product catalogs
  • Not acceptable when: reading account balance after a debit

Q86. What are the main strategies for scaling?

  • Vertical scaling: Bigger machine (CPU, RAM) — simple, has ceiling
  • Horizontal scaling: More machines — requires stateless services, load balancer
  • Read replicas: Scale reads; writes still go to primary
  • Caching: Serve repeated reads from memory
  • CDN: Serve static assets from edge nodes close to users
  • Async / queue: Decouple slow work from the request path
  • Sharding: Partition data across nodes for write scaling

Q87. What is a load balancer and what algorithms do they use?

Routes incoming traffic to a pool of servers.

IP HashHash client IP → sticky serverSession affinity needed
WeightedServers get traffic proportional to weightHeterogeneous capacity

Q88. How would you design for high availability (HA)?

Eliminate single points of failure at every layer:

  • App: Multiple replicas behind a load balancer. Auto-scaling group.
  • Database: Primary + read replica. Multi-AZ with automatic failover (Aurora).
  • Cache: Redis Cluster or Sentinel for failover.
  • Message queue: Kafka with replication factor ≥ 3, min.insync.replicas=2.
  • DNS: Route53 health checks with failover routing.
  • Cross-region: Active-passive or active-active multi-region for disaster recovery.

Q90. What is idempotency and why does it matter in APIs?

An operation is idempotent if calling it multiple times produces the same result as calling it once.

  • GET, PUT, DELETE — idempotent by definition
  • POST — NOT idempotent by default (creates new resource each time)
Pattern: Accept an Idempotency-Key header. Store key + result. On duplicate request, return cached result. 
@PostMapping("/payments")
public PaymentResult pay(@RequestHeader("Idempotency-Key") String key,
                         @RequestBody PaymentRequest req) {
    return idempotencyStore.getOrCompute(key, () -> paymentService.pay(req));
}
Stripe, Adyen, and all major payment APIs require idempotency keys for POST requests.

Q94. How do you propagate trace IDs across microservices?

Q95. What is the Outbox Pattern and why is it important?

Problem: You write to your database AND publish to Kafka in the same operation. If Kafka publish fails after DB commit — your event is lost. If DB commit fails after Kafka publish — you've published a ghost event. Outbox Pattern:
  1. Write your domain entity AND an outbox_events row in the same DB transaction
  2. A separate poller (or Debezium CDC) reads from outbox_events and publishes to Kafka
  3. On successful publish, mark the outbox row as processed
@Transactional
public void placeOrder(Order order) {
    orderRepo.save(order);
    outboxRepo.save(new OutboxEvent("ORDER_PLACED", order.getId(), serialize(order)));
    // Both writes in one transaction — atomically consistent
}
// Separate Debezium connector or @Scheduled job reads outbox → Kafka
Debezium (Change Data Capture) reads Postgres WAL log directly — zero polling latency, no code needed in the publisher.

Q106. What is virtual thread (Project Loom) and why does it matter?

Java 21 — Virtual Threads (Thread.ofVirtual()):
  • Millions of lightweight threads managed by the JVM, not the OS
  • Blocking a virtual thread does NOT block a platform thread — JVM parks it and reuses the carrier thread
Impact on Spring Boot: 
# Spring Boot 3.2+ — enable virtual threads
spring:
  threads:
    virtual:
      enabled: true
  • Tomcat serves each request on a virtual thread → simple blocking code scales like reactive code
  • No need for WebFlux just to avoid blocking — blocking JDBC, RestTemplate calls are fine
Before Loom: 200 Tomcat threads → 200 concurrent requests max (threads blocked on I/O).
After Loom: 200 carrier threads → millions of concurrent virtual threads — I/O blocking is cheap.

Q107. What causes high GC pressure and how do you reduce it?

Causes:
  • Allocating large objects or many short-lived objects in hot paths
  • Caching unbounded collections in heap
  • Autoboxing in tight loops (int → Integer)
Reductions: 
// Avoid autoboxing in hot path
Map<String, Integer> map = new HashMap<>();
map.merge(key, 1, Integer::sum); // creates Integer object per call

// Better: primitive-specialized collections (Eclipse Collections, Koloboke)
MutableObjectIntMap<String> map = ObjectIntMaps.mutable.empty();
map.addToValue(key, 1); // no boxing

// Reuse objects — StringBuilder over String concat in loops
StringBuilder sb = new StringBuilder();
for (String s : list) sb.append(s).append(',');

Q108. What is connection pool exhaustion and how do you detect it?

Symptom: Requests hang or timeout waiting for a DB connection. HikariPool-1 - Connection is not available, request timed out after 30000ms. Detection:
  • Metric: hikaricp.connections.pending > 0
  • Metric: hikaricp.connections.active = maximum-pool-size (pool maxed out)
Root causes:
  1. Slow queries holding connections too long
  2. Pool size too small for traffic
  3. Transaction left open (missing @Transactional or exception swallowed)
  4. Deadlock in DB — connections wait forever for lock release
Fix:
  • Increase maximum-pool-size (carefully — more connections isn't always better)
  • Set connection-timeout to fail fast rather than queue
  • Find and fix slow queries with EXPLAIN ANALYZE
  • Ensure @Transactional methods always commit or rollback

🧬 SECTION 22 — Advanced Java

Q109. What are Java Generics and what is type erasure?

Generics provide compile-time type safety. At runtime, type parameters are erased — the JVM sees only raw types.

List<String> strings = new ArrayList<>();
List<Integer> ints   = new ArrayList<>();
strings.getClass() == ints.getClass(); // true — both are ArrayList at runtime
Implications:
  • Cannot do new T() or T.class — T is not known at runtime
  • Cannot create new ArrayList<T>[] — generic array creation not allowed
  • Wildcards: List<? extends Number> (read-only covariant), List<? super Integer> (write PECS: Producer extends, Consumer super)

Q110. What is reflection and when should you avoid it?

Reflection lets you inspect and manipulate classes, fields, and methods at runtime.

Class<?> clazz = Class.forName("com.example.UserService");
Method method = clazz.getMethod("findById", Long.class);
Object result = method.invoke(serviceInstance, 42L);
Use for: Frameworks (Spring, Jackson, JUnit), serialisation, dependency injection. Avoid for:
  • Performance-critical code — 10–100x slower than direct calls
  • Security-sensitive code — bypasses access modifiers
  • Any place where you control the types — generics or interfaces are cleaner

Q111. What is the difference between Comparable and Comparator?

// Comparable — natural ordering, implemented BY the class
public class Employee implements Comparable<Employee> {
    public int compareTo(Employee other) {
        return this.salary - other.salary; // natural order = by salary
    }
}

// Comparator — external ordering, implemented OUTSIDE the class
Comparator<Employee> byName = Comparator.comparing(Employee::getName);
Comparator<Employee> bySalaryThenName = Comparator
    .comparingInt(Employee::getSalary)
    .thenComparing(Employee::getName);

employees.sort(bySalaryThenName);
Use Comparable for one obvious natural ordering. Use Comparator for multiple orderings or when you don't own the class.

Q115. How do you decide whether to use a synchronous or asynchronous approach?

Use synchronous when:
  • Caller needs the result immediately
  • Operation is fast (< 100ms)
  • Simplicity is more important than throughput
Use asynchronous when:
  • Operation is slow (file upload, email send, third-party API)
  • Caller doesn't need to wait for the result
  • You need to decouple services (avoid cascade failures)
  • You need to buffer against traffic spikes
Rule of thumb: If the operation is not critical to the HTTP response, make it async.

Q116. How do you handle backward compatibility in a REST API?

  1. Never remove or rename existing fields — add new fields only
  2. Version the API — /v1/orders, /v2/orders or Accept: application/vnd.api+json;version=2
  3. Tolerate unknown fields — consumer ignores fields it doesn't know (Postel's Law)
  4. Deprecation lifecycle — mark old endpoint @Deprecated, log usage, set sunset date, notify consumers
  5. Database: Add columns as nullable; never rename or drop columns in a running system

Q117. What is technical debt and how do you manage it without blocking product delivery?

Technical debt: Shortcuts taken for speed that cost more time later to fix. Management strategies:
  • Debt log: Track debt items in the backlog with estimated impact (not a wish list — prioritised by risk)
  • Boy Scout Rule: Leave code slightly better than you found it — refactor incrementally during feature work
  • 20% capacity rule: Reserve ~20% of each sprint for tech debt reduction (negotiate with PM upfront)
  • Make debt visible: Use ADRs to document shortcuts taken and their remediation plan
  • Prefer targeted rewrites over big-bang: Strangle the old code while shipping new features

Q118. How do you approach code review?

As reviewer:
  • Read the PR description first — understand the why
  • Check for correctness first (logic, edge cases), then style
  • Ask questions rather than make demands: "What happens if X is null here?"
  • Approve with minor nits — don't block for style preferences
  • Review tests as seriously as production code
As author:
  • Small PRs (< 400 lines) — easier to review thoroughly
  • Write a PR description explaining the change, not what the diff shows
  • Self-review before requesting — catch the obvious issues yourself
  • Respond to all comments, even if just "acknowledged"

⚛️ SECTION 24 — Reactive Programming (WebFlux & Project Reactor)

Q119. What is reactive programming and when should you use it?

Reactive programming is a paradigm for asynchronous, non-blocking data streams. Instead of blocking a thread while waiting for I/O, the thread is released and a callback is invoked when the result is ready. Use reactive (WebFlux) when:
  • Extremely high concurrency with I/O-bound operations (thousands of concurrent connections)
  • Streaming responses (SSE, WebSocket, large file transfers)
  • You need back-pressure — consumer controls the rate of data flow
Do NOT use WebFlux when:
  • Your team isn't experienced with reactive — debugging async call stacks is hard
  • You use blocking JDBC drivers — mixing blocking code into a reactive pipeline blocks the event loop (catastrophic)
  • Java 21 virtual threads are available — blocking + virtual threads is simpler and nearly as scalable

Q121. What is back-pressure in Project Reactor?

Back-pressure is a mechanism where the subscriber controls how many elements it can handle, preventing the publisher from overwhelming it.

Flux.range(1, 1_000_000)
    .onBackpressureBuffer(1000)    // buffer up to 1000 items
    .publishOn(Schedulers.boundedElastic())
    .subscribe(item -> {
        processSlowly(item);       // slow consumer
    });
In practice, most Spring WebFlux apps don't need manual back-pressure tuning — the framework handles it. The concept matters most when dealing with unbounded streams (Kafka, file reads, WebSocket).

Q122. What is publishOn vs subscribeOn?

// subscribeOn — affects where the SUBSCRIPTION (source) runs
Mono.fromCallable(() -> blockingDbCall())
    .subscribeOn(Schedulers.boundedElastic()); // offload blocking call to elastic pool

// publishOn — affects where DOWNSTREAM operators run
Flux.fromIterable(items)
    .publishOn(Schedulers.parallel())          // process downstream on parallel scheduler
    .map(this::expensiveTransform);
Rule: Use subscribeOn(Schedulers.boundedElastic()) to wrap blocking calls. Use publishOn(Schedulers.parallel()) to move CPU-intensive work to parallel threads.

Q130. Two Pointers — when and how?

Use when array/string is sorted, or when you need pairs/triplets summing to a target.

// Two Sum in sorted array — O(N), O(1) space
public int[] twoSum(int[] nums, int target) {
    int left = 0, right = nums.length - 1;
    while (left < right) {
        int sum = nums[left] + nums[right];
        if (sum == target) return new int[]{left, right};
        if (sum < target) left++;
        else right--;
    }
    return new int[]{};
}

Q131. Sliding Window — when and how?

Use for subarray/substring problems: max/min/sum in a window of size K, or longest substring with constraint.

// Maximum sum subarray of size K — O(N)
public int maxSumSubarray(int[] nums, int k) {
    int windowSum = 0, maxSum = 0;
    for (int i = 0; i < nums.length; i++) {
        windowSum += nums[i];
        if (i >= k) windowSum -= nums[i - k]; // shrink window
        if (i >= k - 1) maxSum = Math.max(maxSum, windowSum);
    }
    return maxSum;
}

// Variable window — longest substring without repeating chars
public int lengthOfLongestSubstring(String s) {
    Map<Character, Integer> seen = new HashMap<>();
    int max = 0, left = 0;
    for (int right = 0; right < s.length(); right++) {
        char c = s.charAt(right);
        if (seen.containsKey(c)) left = Math.max(left, seen.get(c) + 1);
        seen.put(c, right);
        max = Math.max(max, right - left + 1);
    }
    return max;
}

// BFS — shortest path in unweighted graph
public int bfs(int[][] grid, int[] start, int[] end) {
    Queue<int[]> queue = new LinkedList<>();
    Set<String> visited = new HashSet<>();
    queue.offer(start);
    int steps = 0;
    int[][] dirs = {{0,1},{0,-1},{1,0},{-1,0}};
    while (!queue.isEmpty()) {
        int size = queue.size();
        while (size-- > 0) {
            int[] curr = queue.poll();
            if (Arrays.equals(curr, end)) return steps;
            for (int[] d : dirs) {
                int nr = curr[0]+d[0], nc = curr[1]+d[1];
                String key = nr+","+nc;
                if (nr>=0 && nr<grid.length && nc>=0 && nc<grid[0].length
                    && grid[nr][nc]==0 && visited.add(key))
                    queue.offer(new int[]{nr, nc});
            }
        }
        steps++;
    }
    return -1;
}

Q133. Dynamic Programming — recognising the pattern

DP applies when: overlapping subproblems + optimal substructure.

Recognition signals:
  • "Maximum/minimum number of..."
  • "Number of ways to..."
  • "Can you reach / achieve..."
// Fibonacci — memoization (top-down)
Map<Integer, Long> memo = new HashMap<>();
long fib(int n) {
    if (n <= 1) return n;
    return memo.computeIfAbsent(n, k -> fib(k-1) + fib(k-2));
}

// Coin change — tabulation (bottom-up) — O(amount * coins)
public int coinChange(int[] coins, int amount) {
    int[] dp = new int[amount + 1];
    Arrays.fill(dp, amount + 1); // "infinity"
    dp[0] = 0;
    for (int i = 1; i <= amount; i++)
        for (int coin : coins)
            if (coin <= i) dp[i] = Math.min(dp[i], dp[i - coin] + 1);
    return dp[amount] > amount ? -1 : dp[amount];
}

Q134. What are Window Functions and when do you use them?

Window functions compute values across a set of rows related to the current row without collapsing them (unlike GROUP BY).

-- Row number per group
SELECT
    user_id,
    order_id,
    amount,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at DESC) AS rn
FROM orders;

-- Running total
SELECT
    order_id,
    amount,
    SUM(amount) OVER (PARTITION BY user_id ORDER BY created_at) AS running_total
FROM orders;

-- Rank with ties
SELECT name, salary,
    RANK()       OVER (ORDER BY salary DESC) AS rank,
    DENSE_RANK() OVER (ORDER BY salary DESC) AS dense_rank
FROM employees;

-- Get latest order per user (common pattern)
SELECT * FROM (
    SELECT *, ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at DESC) AS rn
    FROM orders
) t WHERE rn = 1;

🏆 SECTION 29 — CQRS & Event Sourcing

Q136. What is CQRS?

Command Query Responsibility Segregation — separate the write model (commands) from the read model (queries). 
Write Side:  POST /orders  →  OrderCommandService  →  orders DB (normalised)
                                                    →  publishes OrderCreated event
Read Side:   GET /orders   →  OrderQueryService    →  orders_view DB (denormalised, optimised for reads)
                                                        ← materialised from events
Benefits:
  • Read model can be a completely different DB (Elasticsearch for full-text, Redis for speed)
  • Scale reads and writes independently
  • Read model can be rebuilt from events if corrupted
When NOT to use: Simple CRUD apps — CQRS adds significant complexity for no benefit.

Q137. What is Event Sourcing?

Instead of storing current state, store every event that led to the current state.

// Traditional: store current balance
Account { balance: 950 }

// Event Sourced: store events
AccountOpened    { amount: 1000 }
MoneyWithdrawn   { amount: 50 }
// Current state = replay all events → balance = 950
Benefits:
  • Complete audit log — answer "what was the balance on March 15?" by replaying to that date
  • Replay events to build new read models
  • Temporal queries, debugging, compliance
Challenges:
  • Event schema evolution is hard — old events must still be replayable
  • Rebuilding state requires replaying all events (snapshots mitigate this)
  • Conceptually complex for teams new to it

🔐 SECTION 33 — OAuth2 & OIDC (Common in Sony/Enterprise Contexts)

Q145. Explain the OAuth2 Authorization Code Flow.

1. User clicks "Login" → App redirects to Auth Server with:
   client_id, redirect_uri, scope, response_type=code, state

2. User authenticates at Auth Server → Auth Server redirects back with:
   ?code=AUTH_CODE&state=...

3. App backend exchanges code for tokens (server-to-server):
   POST /token  { code, client_id, client_secret, redirect_uri }
   Response: { access_token, refresh_token, id_token, expires_in }

4. App uses access_token to call Resource Server (your API)

5. Resource Server validates token (JWT signature or introspection)
Why code flow? The access_token never touches the browser URL bar. The exchange happens server-to-server with client_secret. PKCE extends this for SPAs/mobile where there's no server secret.

♻️ SECTION 34 — Distributed Locking (Used in Sony/Valiant Projects)

Q147. What is distributed locking and when do you need it?

Problem: Multiple instances of your service run in parallel. You need to ensure only ONE instance executes a critical section at a time. Use cases:
  • Scheduled jobs (only one pod should run the 2 AM payroll job)
  • Inventory reservation (prevent overselling)
  • Idempotent processing with state machines
Bad solution: Database SELECT FOR UPDATE — works but creates DB contention. Good solution: Redis-based lock (Redisson): 
RLock lock = redissonClient.getLock("payroll-job-lock");
boolean acquired = lock.tryLock(0, 30, TimeUnit.SECONDS); // wait 0s, hold max 30s
if (acquired) {
    try {
        runPayrollJob();
    } finally {
        lock.unlock();
    }
} else {
    log.info("Another instance is running the job — skipping");
}
Redlock algorithm: Acquire lock on majority of N Redis nodes — more fault-tolerant than single-node lock. Controversial (see Martin Kleppmann vs Antirez debate) — for most use cases single-node is sufficient with proper TTL.

Q151. Explain Orchestration vs Choreography Saga with a real example from your CV.

  AccountService catches DebitFailedEvent → publishes FundsUnreservedEvent

@SagaOrchestrator
public class FundTransferSaga {
    public void start(TransferCommand cmd) {
        send(new ReserveFundsCommand(cmd.getSourceAccountId(), cmd.getAmount()));
    }

    @SagaEventHandler
    public void on(FundsReservedEvent e) {
        send(new DebitAccountCommand(e.getAccountId(), e.getAmount()));
    }

    @SagaEventHandler
    public void on(DebitFailedEvent e) {
        send(new ReleaseReservationCommand(e.getAccountId())); // compensate

    public void on(DebitSucceededEvent e) {
        send(new CreditTargetCommand(targetAccountId, e.getAmount()));

🔬 SECTION 39 — Hibernate & JPA Deep Dive

Q156. What is the Hibernate L1 vs L2 cache?

L1 Cache (Session/EntityManager cache):
  • Automatically enabled — always active within one transaction
  • Within the same Session, loading the same entity twice returns the cached instance
  • Flushed when Session closes
// L1 cache in action
User u1 = em.find(User.class, 1L); // DB query
User u2 = em.find(User.class, 1L); // NO DB query — returned from L1 cache
assert u1 == u2;                   // same object reference
L2 Cache (cross-session, optional):
  • Shared across Sessions/transactions
  • Backed by Ehcache, Hazelcast, or Redis
  • Explicitly enabled per entity
@Entity
@Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
public class ProductCatalog { ... }

spring.jpa.properties:
  hibernate.cache.use_second_level_cache: true
  hibernate.cache.region.factory_class: org.hibernate.cache.jcache.JCacheRegionFactory
Warning: L2 cache is tricky with distributed deployments — if Service A updates an entity, Service B's L2 cache still has the stale version. Use L2 only for read-mostly, low-update data (e.g., country lookup tables).

Q157. What is the difference between FetchType.LAZY proxy and an actual entity?

When you load an entity with a @ManyToOne(fetch=LAZY) relationship, Hibernate injects a proxy object — a subclass of the related entity with all fields null until accessed.

Order order = orderRepo.findById(1L).get(); // loads Order only
Customer c = order.getCustomer();           // returns PROXY — no DB hit yet
String name = c.getName();                  // NOW hits DB to initialise proxy
LazyInitializationException: Accessing the proxy AFTER the Session closes. 
// BROKEN — transaction closed before accessing lazy field
Order order = orderRepo.findById(1L).get(); // @Transactional method returns
String name = order.getCustomer().getName(); // Session closed → EXCEPTION
Fix: Use @Transactional at the service layer (not repository), or use JOIN FETCH.

@Bean
public Step processEmployees(StepBuilderFactory steps,
        ItemReader<Employee> reader,
        ItemProcessor<Employee, Payslip> processor,

How we applied it at Valiant:

  2. Started with Notifications (lowest risk, self-contained)
  3. Deployed Notification Service behind Kong API Gateway

Q160. What is the Anti-Corruption Layer (ACL)?

When integrating with a legacy system or third-party API, the ACL is a translation layer that prevents the external system's model from polluting your domain model.

// Sony's external device management API returns their format
public class SonyDeviceApiClient {
    public SonyDeviceResponse getDevice(String deviceId) { ... }
}

// ACL translates Sony's model to your domain model
@Component
public class DeviceAntiCorruptionLayer {
    private final SonyDeviceApiClient sonyClient;

    public Device getDevice(DeviceId id) {
        SonyDeviceResponse response = sonyClient.getDevice(id.value());
        return Device.builder()          // your domain model
            .id(DeviceId.of(response.getDevId()))
            .model(response.getModelName())
            .registeredAt(Instant.ofEpochMilli(response.getRegTs()))
            .build();
    }
}
Without ACL: Sony's SonyDeviceResponse leaks into your service layer. When Sony changes their API, you rewrite your entire codebase. With ACL: one class changes.

// MDC (Mapped Diagnostic Context) — thread-local key-value for logging
@Component
public class TraceIdFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest req,
                                    HttpServletResponse res,
                                    FilterChain chain) throws IOException, ServletException {
        // Extract from incoming request or generate new
        String traceId = Optional.ofNullable(req.getHeader("X-Trace-Id"))
            .orElse(UUID.randomUUID().toString());

        MDC.put("traceId", traceId);
        res.setHeader("X-Trace-Id", traceId);
        try {
            chain.doFilter(req, res);
        } finally {
            MDC.clear();  // must clear to prevent ThreadLocal leak in thread pool
        }
    }
}

Q163. You migrated a monolith at Axon Active. What's the hardest technical problem in that migration?

SECTION 46 — API Design Best Practices

Q171. What are the REST API pagination strategies?

Offset pagination: 
GET /orders?page=2&size=20
  • Simple to implement
  • Problem: Inconsistent with inserts/deletes (page drift), slow for large offsets (OFFSET 10000 scans 10k rows)
Cursor pagination: 
GET /orders?cursor=eyJpZCI6MTAwfQ&size=20
# Response: { data: [...], nextCursor: "eyJpZCI6MTIwfQ" }
  • Cursor = encoded last-seen value (e.g., base64 of {"id": 100})
  • Stable — inserts/deletes don't affect pages
  • Fast — WHERE id > 100 ORDER BY id LIMIT 20 uses index
  • Problem: No random page access ("go to page 5")
Best practice: Use cursor pagination for feeds, timelines, large datasets. Use offset for admin tables where random page access is needed.

Q174. What are the three multi-tenancy data isolation strategies?

SECTION 48 — Deployment Strategies

Q176. What is the difference between Blue-Green and Canary deployments?

Blue-Green: 
Blue (v1) ← 100% traffic     Green (v2) ← 0% traffic
                  ↓ switch
Blue (v1) ← 0% traffic      Green (v2) ← 100% traffic
  • Instant cutover — one DNS/load balancer change
  • Full rollback in seconds (flip back to Blue)
  • Cost: Requires double infrastructure during transition
  • Risk: 100% of traffic hits new version immediately after switch
Canary: 
v1 ← 95% traffic
v2 ←  5% traffic  (canary — like the "canary in a coal mine")
v2 ← 50% traffic  (gradually increase if metrics look good)
v2 ← 100% traffic
  • Gradual rollout — limits blast radius
  • Requires traffic splitting (ALB weighted routing, Istio, Argo Rollouts)
  • Best for: High-traffic systems where even 5% of traffic catching a bug is manageable

Q183. What is table partitioning in Postgres?

Split a large table into smaller physical partitions while appearing as one logical table.

-- Range partitioning by date (common for time-series / audit logs)
CREATE TABLE audit_log (
    id BIGSERIAL,
    event_type VARCHAR(100),
    created_at TIMESTAMP NOT NULL
) PARTITION BY RANGE (created_at);

CREATE TABLE audit_log_2024 PARTITION OF audit_log
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

CREATE TABLE audit_log_2025 PARTITION OF audit_log
    FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');

-- Queries with partition key in WHERE clause only scan relevant partition
SELECT * FROM audit_log
WHERE created_at >= '2025-03-01';  -- only reads audit_log_2025
Benefits: Faster queries (partition pruning), faster deletes (drop old partition = instant), better vacuum performance.

SECTION 58 — Thread Pool Design

Q193. How do you size a thread pool correctly?

CPU-bound tasks: pool size = number of CPU cores

More threads than cores → context switching overhead, no benefit.

I/O-bound tasks: pool size = CPU cores × (1 + wait_time / compute_time)

If a thread spends 90% of time waiting on I/O: pool = 8 × (1 + 9) = 80 threads

// CPU-bound: image processing, encryption
ExecutorService cpuPool = Executors.newFixedThreadPool(
    Runtime.getRuntime().availableProcessors());

// I/O-bound: DB calls, HTTP calls
ExecutorService ioPool = Executors.newFixedThreadPool(
    Runtime.getRuntime().availableProcessors() * 10);

// Prefer: bounded thread pool with rejection policy
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    10,    // corePoolSize
    50,    // maximumPoolSize
    60L, TimeUnit.SECONDS,   // keepAlive for threads above core
    new ArrayBlockingQueue<>(1000),  // bounded queue — prevents OOM
    new ThreadPoolExecutor.CallerRunsPolicy()  // backpressure: caller runs task
);

Q198. What is JWT token best practice for access vs refresh tokens?

Access Token:
- Short-lived: 15 minutes
- Stateless JWT — validated by signature only
- Contains: userId, roles, scopes, exp
- Never store in localStorage (XSS risk) → use memory or secure cookie

Refresh Token:
- Long-lived: 7–30 days
- Stored in HttpOnly, Secure, SameSite=Strict cookie (not accessible by JS)
- Stored server-side (Redis or DB) for revocation
- On use: issue new access token + rotate refresh token (single use)

Q200. What is the Law of Demeter and how do you apply it?

"Talk to your friends, not strangers." A method should only call methods on:

  • Its own class (this)
  • Its direct dependencies (constructor/method params)
  • Objects it creates
  • NOT: chains of method calls

String city = order.getCustomerCity();

SECTION 73 — Database Transactions Deep Dive

Q220. What are database savepoints and when do you use them?

A savepoint marks a position within a transaction — you can rollback to it without rolling back the entire transaction.

// Spring — NESTED propagation uses savepoints internally
@Transactional(propagation = Propagation.NESTED)
public void processOrderItem(OrderItem item) {
    // If THIS fails, only this item's changes rollback
    // The outer transaction (processOrder) continues with other items
    itemRepo.save(item);
    inventoryService.reserve(item);
}

@Transactional
public void processOrder(Order order) {
    orderRepo.save(order);
    for (OrderItem item : order.getItems()) {
        try {
            processOrderItem(item);   // NESTED — savepoint before this call
        } catch (InsufficientStockException e) {
            item.setStatus("BACKORDERED");  // handle gracefully, continue
        }
    }
}
NESTED requires: Database support for savepoints (Postgres ✅, MySQL ✅, H2 ✅) AND a DataSourceTransactionManager (not JTA).

Q221. What is @Transactional on a private method — does it work?

No. Spring AOP proxy cannot intercept private methods — the proxy can only override public methods. 
@Service
public class OrderService {

    public void placeOrder(Order order) {
        saveOrder(order);     // calls this.saveOrder() — bypasses proxy!
    }

    @Transactional         // IGNORED — proxy never intercepts this
    private void saveOrder(Order order) {
        orderRepo.save(order);
    }
}

// FIX: make the method public, OR extract to a separate @Service bean
@Service
public class OrderPersistenceService {
    @Transactional
    public void saveOrder(Order order) {  // public — proxy intercepts correctly

Q222. What is the difference between requests and limits in Kubernetes?

    memory: "256Mi"     # K8s won't place pod on a node with < 256Mi available

SECTION 76 — Performance Anti-Patterns

Q225. What are the most common performance anti-patterns in Spring Boot apps?

1. N+1 queries in a loop: 
// BAD — N+1
List<Order> orders = orderRepo.findAll();
orders.forEach(o -> log.info(o.getCustomer().getName())); // N DB hits
// FIX: JOIN FETCH or @EntityGraph
2. Loading entire table into memory: 
// BAD — loads ALL records
List<User> all = userRepo.findAll();
// FIX: use pagination or Stream
@Query("SELECT u FROM User u")
@QueryHints(@QueryHint(name = HINT_FETCH_SIZE, value = "50"))
Stream<User> streamAll();  // fetches in chunks, don't load all in memory
3. Synchronous HTTP calls in a loop: 
// BAD — serial HTTP calls, O(N) latency
items.forEach(item -> price = pricingService.getPrice(item)); // sequential

// GOOD — parallel CompletableFuture
List<CompletableFuture<Price>> futures = items.stream()
    .map(item -> CompletableFuture.supplyAsync(() -> pricingService.getPrice(item)))
    .toList();
CompletableFuture.allOf(futures.toArray(new CompletableFuture[0])).join();
4. Unnecessary serialisation/deserialisation: 
// BAD — convert to String, parse back in same JVM
String json = objectMapper.writeValueAsString(order);
Order copy = objectMapper.readValue(json, Order.class); // expensive deep copy!
// GOOD: use a proper copy mechanism or redesign to avoid the copy

Q226. What is the difference between @Async and CompletableFuture.supplyAsync()?

// @Async — Spring manages the executor, method runs asynchronously
// Caller gets Future/CompletableFuture back immediately
@Async("customExecutor")
public CompletableFuture<String> fetchAsync(String id) {
    return CompletableFuture.completedFuture(doFetch(id));
}

// CompletableFuture.supplyAsync() — explicit, no Spring proxy needed
CompletableFuture<String> future = CompletableFuture
    .supplyAsync(() -> doFetch(id), customExecutor);
Key differences:
  • @Async requires @EnableAsync and Spring proxy — won't work on private methods or self-calls
  • @Async exception handling is tricky — use AsyncUncaughtExceptionHandler
  • CompletableFuture.supplyAsync() is explicit — easier to reason about, chain, compose
  • Prefer CompletableFuture directly for complex async chains; use @Async for simple fire-and-forget

        }
    }
}

SECTION 82 — Java Concurrency Advanced

Q235. What is StampedLock and when is it better than ReentrantReadWriteLock?

// ReentrantReadWriteLock — multiple readers OR one writer
ReadWriteLock rwLock = new ReentrantReadWriteLock();
// Problem: write lock is exclusive — readers must wait even if write is rare

// StampedLock (Java 8+) — optimistic reads avoid locking entirely
StampedLock lock = new StampedLock();

// Optimistic read — NO lock acquired, just a stamp
double read() {
    long stamp = lock.tryOptimisticRead();
    double currentValue = this.value;         // read without lock
    if (!lock.validate(stamp)) {              // check if a write occurred
        stamp = lock.readLock();              // fall back to real read lock
        try { currentValue = this.value; }
        finally { lock.unlockRead(stamp); }
    }
    return currentValue;
}

void write(double newValue) {
    long stamp = lock.writeLock();
    try { this.value = newValue; }
    finally { lock.unlockWrite(stamp); }
}
Use StampedLock when: Very high read contention, writes are rare. The optimistic read path has near-zero overhead — no lock acquisition, no cache coherence traffic.

Q238. How do you use S3 efficiently for large file processing?

// Multipart upload for large files (> 100MB)
@Service
public class S3UploadService {

    private final S3Client s3;

    public String uploadLargeFile(String bucket, String key, InputStream data, long size) {
        CreateMultipartUploadResponse init = s3.createMultipartUpload(
            r -> r.bucket(bucket).key(key).contentType("application/octet-stream"));

        String uploadId = init.uploadId();
        List<CompletedPart> parts = new ArrayList<>();
        byte[] buffer = new byte[10 * 1024 * 1024]; // 10MB parts
        int partNumber = 1;

        try {
            int bytesRead;
            while ((bytesRead = data.read(buffer)) > 0) {
                UploadPartResponse part = s3.uploadPart(
                    r -> r.bucket(bucket).key(key).uploadId(uploadId)
                           .partNumber(partNumber),
                    RequestBody.fromBytes(Arrays.copyOf(buffer, bytesRead)));
                parts.add(CompletedPart.builder()
                    .partNumber(partNumber++).eTag(part.eTag()).build());
            }

            s3.completeMultipartUpload(r -> r.bucket(bucket).key(key)
                .uploadId(uploadId)
                .multipartUpload(m -> m.parts(parts)));

            return "s3://" + bucket + "/" + key;
        } catch (Exception e) {
            s3.abortMultipartUpload(r -> r.bucket(bucket).key(key).uploadId(uploadId));
            throw new RuntimeException("Upload failed", e);
        }
    }
}

Q240. What is mutation testing and why does it matter?

Mutation testing evaluates test quality by introducing small bugs (mutations) into production code and checking if your tests catch them. 
Original:  if (balance >= amount) {
Mutation:  if (balance > amount) {   ← mutant (edge case: balance == amount)
           if (balance <= amount) {  ← mutant (inverted condition)

If your test suite doesn't catch these mutations → tests have gaps in coverage.

<plugin>

SECTION 86 — DDD (Domain-Driven Design)

Q242. What is an Aggregate in DDD and why does it matter for microservices?

An Aggregate is a cluster of domain objects treated as a unit for data changes. One object is the Aggregate Root — the only entry point.

// Order aggregate — Order is the root
@Entity
public class Order {   // Aggregate Root
    private OrderId id;
    private OrderStatus status;
    private List<OrderItem> items;   // items are part of this aggregate
    private Money total;

    // All mutations go through the root — enforces invariants
    public void addItem(Product product, int quantity) {
        if (status != OrderStatus.DRAFT) {
            throw new IllegalStateException("Cannot modify a confirmed order");
        }
        items.add(new OrderItem(product.getId(), product.getPrice(), quantity));
        this.total = calculateTotal();
    }

    public void confirm() {
        if (items.isEmpty()) throw new IllegalStateException("Cannot confirm empty order");
        this.status = OrderStatus.CONFIRMED;
        registerEvent(new OrderConfirmedEvent(this.id));  // domain event
    }
}
Rules:
  • Only reference other aggregates by ID (not by object reference)
  • One repository per aggregate root
  • Transactions should not span multiple aggregates — use domain events + Saga for cross-aggregate consistency

Q245. What is Server-Side Request Forgery (SSRF) and how do you prevent it?

SSRF: Attacker tricks your server into making HTTP requests to internal systems. 
Attacker sends: POST /api/fetch-url { "url": "http://169.254.169.254/latest/meta-data/" }
Your server fetches it → returns AWS instance metadata → attacker gets IAM credentials
Prevention: 
@Component
public class SafeUrlValidator {
    private static final Set<String> ALLOWED_HOSTS = Set.of("api.partner.com", "cdn.example.com");

    public void validate(String url) throws MalformedURLException {
        URL parsed = new URL(url);

        // Block private IP ranges
        InetAddress addr = InetAddress.getByName(parsed.getHost());
        if (addr.isLoopbackAddress() || addr.isSiteLocalAddress() || addr.isLinkLocalAddress()) {
            throw new SecurityException("Requests to private IP ranges are not allowed");
        }

        // Allowlist approach — only permitted hosts
        if (!ALLOWED_HOSTS.contains(parsed.getHost())) {
            throw new SecurityException("Host not in allowlist: " + parsed.getHost());
        }
    }
}

SECTION 89 — Postgres Deep Dive

Q247. What is VACUUM in Postgres and why does it matter?

MVCC (Multi-Version Concurrency Control): Postgres doesn't overwrite rows on UPDATE/DELETE — it creates new versions and marks old ones as "dead". VACUUM reclaims dead tuples:
  • AUTOVACUUM: Runs automatically in the background — tracks tables with many dead tuples
  • VACUUM ANALYZE: Reclaims space AND updates planner statistics
  • VACUUM FULL: Rewrites the entire table (exclusive lock, not for production)
-- Check table bloat (dead tuples accumulating)
SELECT relname, n_live_tup, n_dead_tup,
       round(n_dead_tup * 100.0 / NULLIF(n_live_tup + n_dead_tup, 0), 2) AS dead_pct
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC;

-- Manually trigger vacuum (non-blocking)
VACUUM ANALYZE orders;

-- Check autovacuum is keeping up
SELECT schemaname, tablename, last_autovacuum, autovacuum_count
FROM pg_stat_user_tables
WHERE last_autovacuum < NOW() - INTERVAL '1 day'
ORDER BY n_dead_tup DESC;
Symptom of vacuum lag: Table scans get slower, index bloat grows, pg_stat_user_tables.n_dead_tup is high. Fix: tune autovacuum_vacuum_scale_factor (default 0.2 = vacuum when 20% of rows are dead — too high for large tables, lower to 0.01).

Q251. What are Java-specific pitfalls seniors must know?

// 1. Integer cache trap
Integer a = 127, b = 127; a == b;  // true (cached)
Integer c = 128, d = 128; c == d;  // false (new objects)
// Always use .equals() for object comparison

// 2. String.format vs + in loops
String result = "";
for (int i = 0; i < 10000; i++) result += i;  // O(N²) — creates new String each time
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 10000; i++) sb.append(i); // O(N)

// 3. ConcurrentModificationException
List<Order> orders = new ArrayList<>(getOrders());
for (Order o : orders) {
    if (o.isExpired()) orders.remove(o);  // ConcurrentModificationException!
}
orders.removeIf(Order::isExpired);  // correct

// 4. Double for money
double price = 0.1 + 0.2;  // 0.30000000000000004!
BigDecimal price = new BigDecimal("0.1").add(new BigDecimal("0.2")); // 0.3

// 5. Catching Exception too broadly
try { riskyOperation(); }
catch (Exception e) { log.error("Failed", e); }  // swallows RuntimeException!
// Better: catch only what you can handle meaningfully

Q252. What are your principles for designing a good REST API?

1. Use nouns, not verbs in URLs: 
❌ GET  /getOrders
❌ POST /createOrder
✅ GET  /orders
✅ POST /orders
2. Use HTTP methods semantically:

Q253. What makes an API "developer-friendly"?

From your experience building Sony Global APIs:

SECTION 93 — Microservices Best Practices

Q254. What are your rules for designing microservices?

1. Service granularity — not too small, not too big: 
Rule of thumb: a service should be ownable by one small team (2-pizza rule).
Too small: "User first name service" — network overhead, no benefit
Too big:   "User everything service" — back to monolith
Right size: "Identity service" — auth, profile, preferences for one bounded context
2. Own your data — no shared databases: 
✅ Each service has its own database schema
✅ Cross-service data access via API calls
✅ Eventual consistency across services via events

❌ Service A directly queries Service B's table
❌ Two services write to the same table
3. Design for failure — every external call can fail: 
// Every downstream call must have:
- Timeout (connect + read)
- Retry with backoff (for transient failures)
- Circuit breaker (for sustained failures)
- Fallback (graceful degradation)
4. Async by default for cross-service side effects: 
When Order is placed:
  Sync:  → charge payment (user waiting for result)
  Async: → send confirmation email (user doesn't wait)
  Async: → update inventory (user doesn't wait)
  Async: → notify analytics (user definitely doesn't wait)
5. Idempotent consumers: 
Every message consumer must handle duplicate delivery.
Primary key or unique constraint is your last line of defence.

Q255. What is the "strangler fig" vs "big bang" rewrite principle?

From your Axon Active experience:

Big bang rewrite (avoid): 
Month 1-12: Stop all features, rewrite from scratch
Month 13: Launch new system
Reality: Never finishes, scope creep, old bugs + new bugs
Strangler fig (proven): 
Sprint 1: Extract Notification module → new service, proxy traffic
Sprint 2: Old Notification code still there but dead — delete it
Sprint 3: Extract Reporting module → new service...
Sprint N: Monolith is empty — retire it

Risk: Each step is small, always shippable, rollback is easy
Rules for successful strangler:
  1. Start with the least risky module (not the most critical)
  2. Use an API gateway or facade to proxy traffic — clients see no change
  3. Run old and new in parallel briefly — compare outputs for correctness
  4. Only delete old code after new code is proven in production

SECTION 95 — Database Best Practices

Q257. What are your database design principles?

1. Name things consistently: 
-- Tables: plural snake_case
CREATE TABLE orders ( ... );
CREATE TABLE order_items ( ... );

-- Foreign keys: {table}_{column}
order_items.order_id → orders.id

-- Indexes: idx_{table}_{columns}
CREATE INDEX idx_orders_user_id ON orders(user_id);
CREATE INDEX idx_orders_status_created ON orders(status, created_at DESC);
2. Always include audit columns: 
CREATE TABLE orders (
    id          BIGSERIAL PRIMARY KEY,
    -- business columns --
    created_at  TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    created_by  VARCHAR(100) NOT NULL,  -- user or service that created
    version     INTEGER NOT NULL DEFAULT 0  -- for optimistic locking
);
3. Never use SELECT * in production queries: 
// Bad — fetches all columns including LOBs, breaks if schema changes
@Query("SELECT * FROM orders WHERE user_id = ?1")

// Good — explicit columns, deterministic, JPA projections friendly
@Query("SELECT o.id, o.status, o.amount, o.createdAt FROM Order o WHERE o.userId = ?1")
4. Index strategy: 
Index every foreign key (Postgres doesn't auto-index FK columns)
Index columns used in WHERE clauses of frequent queries
Composite index: put most selective column first
Partial index: only index the subset you query (e.g., WHERE status = 'PENDING')
Never index low-cardinality columns alone (e.g., boolean, gender)
5. Migration safety rules: 
✅ Add nullable columns (safe, no lock)
✅ Add new tables (safe)
✅ Add indexes CONCURRENTLY (no table lock)
✅ Add new NOT NULL column: nullable first → backfill → add constraint

❌ Rename columns (breaks running code in rolling deployments)
❌ Drop columns (coordinate with all consumers first)
❌ Change column type (may lock, may lose data)
❌ Remove NOT NULL without default (app may start sending nulls)

SECTION 96 — Release Engineering Principles

Q258. What is trunk-based development and why do you prefer it?

Feature branches (GitFlow): Long-lived feature branches → merge conflicts, integration hell, slow feedback. Trunk-based development:
  • All developers commit to main (trunk) at least daily
  • Features that aren't ready are hidden behind feature flags
  • CI runs on every commit — must stay green always
  • Deploy to production multiple times per day
Trunk-based workflow:
1. Pull latest main
2. Make small change (< 1 day of work)
3. Write test
4. Push to main → CI runs
5. CI green → auto-deploy to staging
6. Manual approval → deploy to prod (or auto if coverage > threshold)
Benefits:
  • No merge conflicts — small, frequent integrations
  • Bugs found immediately — broken main = everyone's problem
  • Deploy anytime — code is always in a deployable state

Q259. What is your approach to hotfixes in production?

Principle: A hotfix is just another deployment with higher urgency.

Process:
1. Create branch from CURRENT PRODUCTION TAG (not main — main may have unreleased work)
2. Apply minimal fix (solve the problem, don't refactor)
3. Write a test that proves the bug
4. Deploy via same pipeline with expedited approval
5. Merge back to main immediately after
6. Write post-mortem within 24 hours

Anti-patterns to avoid:
- Applying fix directly to prod without CI (fix the pipeline, not bypass it)
- Adding multiple changes to a hotfix ("since we're in there anyway")
- Not writing a test (same bug will return)

Q261. What is defence in depth?

Multiple layers of security — compromise of one layer doesn't mean compromise of everything.

Layer 1: Network    → VPC, security groups, private subnets, WAF
Layer 2: Transport  → TLS everywhere, mTLS between services
Layer 3: Auth       → JWT validation, OAuth2 scopes
Layer 4: Authz      → RBAC at API level (@PreAuthorize)

Layer 7: Monitoring → Anomaly detection, CloudTrail, GuardDuty
Sony application: WAF blocked malicious requests before they hit the ALB. Even if WAF was bypassed, API Gateway validated JWT. Even if JWT was forged, service-level authorization checked roles. Even if auth was bypassed, data was encrypted at rest.

SECTION 98 — Performance Principles

Q262. What is your framework for diagnosing a performance problem?

1. MEASURE before optimising — never guess
   Tools: async-profiler flamegraph, JFR, pg_stat_statements

2. Identify the bottleneck layer:
   CPU bound?    → flamegraph shows hot methods
   I/O bound?    → thread dump shows threads WAITING (not RUNNING)
   DB bound?     → slow query log, pg_stat_statements
   Network bound?→ high latency to external service, check with curl timing

3. Fix the highest impact, lowest risk change first:
   - Add missing index → 10x query speedup, zero code change
   - Fix N+1 query → JOIN FETCH or @EntityGraph
   - Add caching → reduce DB load by 80%
   Only then consider: algorithm change, infrastructure scaling

4. Measure after fix — confirm improvement
   Before: P99 = 2000ms, error rate = 5%
   After:  P99 = 180ms, error rate = 0.1%

5. Document the finding
   "Added index on orders(user_id) — reduced GET /orders P99 from 2s to 180ms"

Q263. What is Amdahl's Law and why does it matter for scaling?

Amdahl's Law: The speedup from parallelising a program is limited by the sequential part. 
If 95% of work is parallelisable, 5% is sequential:
  With 2 processors:  speedup = 1/(0.05 + 0.95/2) ≈ 1.9x
  With 10 processors: speedup = 1/(0.05 + 0.95/10) ≈ 6.7x
  With ∞ processors:  speedup = 1/0.05 = 20x (ceiling!)

The 5% sequential code is the ceiling regardless of how many servers you add.
Practical implication:
  • Identify the bottleneck — if it's a single-threaded lock, adding pods won't help
  • At Sony: payroll batch had a single-threaded "total calculation" step — all parallelisation gains were lost there. Fixed by parallelising the aggregation too.
  • The right question: Before buying more servers, ask "what is the sequential bottleneck?"

Q269. What are the production considerations for running Debezium?

1. Replication slot management:
   - WAL accumulates if Debezium lags → disk fills up → DB crash
   - Monitor: SELECT * FROM pg_replication_slots; — check lag_bytes
   - Alert if lag > 1GB

2. Schema changes:
   - Debezium handles most ALTER TABLE automatically
   - Adding/dropping columns: safe if consumer is tolerant (use Avro Schema Registry)
   - Renaming columns: breaks everything — use add-then-drop pattern

3. Initial snapshot:
   - On first start, Debezium snapshots the full table → may take hours for large tables
   - Use 'snapshot.mode=never' if table already exists in target
   - Or use 'snapshot.mode=exported' with pg_dump + Debezium from that LSN

4. Ordering guarantee:
   - Events for the same row → same Kafka partition (keyed by primary key)
   - Order preserved per row — not across tables

SECTION 103 — Architecture Decision Records (ADRs)

Q274. What is an ADR and how do you write one?

An Architecture Decision Record documents a significant architectural decision: the context, the options considered, and the rationale.

# ADR-0042: Use Kafka for device sync events instead of HTTP callbacks

**Date:** 2025-03-15
**Status:** Accepted
**Deciders:** Tuong (Tech Lead), John (Architect), Sarah (PM)

## Context
Sony's device management system needs to notify 50+ downstream services
when a device configuration changes. Current approach: HTTP callbacks to
each service synchronously — causing cascading failures when any service
is slow or down.

## Decision Drivers
- Decouple producers from consumers (services can be down temporarily)
- Support new consumers without changing producer
- Handle 50,000 device updates/minute at peak
- Audit trail of all configuration changes

## Options Considered

### Option 1: Kafka (chosen)
Pros: Decoupled, replay, multiple consumer groups, high throughput
Cons: Operational complexity, requires Kafka cluster management

### Option 2: Continue with HTTP callbacks
Pros: Simple, synchronous confirmation
Cons: Tight coupling, cascading failures, O(N) calls per event

### Option 3: AWS SNS/SQS fan-out
Pros: Managed, no operational overhead
Cons: Vendor lock-in, 7-day retention limit, no replay beyond retention

Use Kafka with MSK (managed). Consumer groups per downstream service.
Retention: 30 days (Sony compliance requirement for audit trail).

## Consequences
- All downstream services must implement Kafka consumers
- Need Debezium or Outbox pattern for atomic DB+Kafka writes

⚡ CACHING CHEATSHEET

When to Cache

✅ Cache when:
  - Same data read repeatedly (product catalog, user profile)
  - Computation is expensive (recommendation results)
  - External API has rate limits or cost per call
  - Database queries are slow and data changes infrequently

❌ Don't cache when:
  - Data changes frequently (stock prices, live scores)
  - Data is user-specific and volume is huge (personalised feeds)
  - Data is sensitive (PII — avoid in shared cache without encryption)
  - Consistency is critical (account balance after debit)

Cache Invalidation Strategies

1. TTL (Time-To-Live):
   Simple, no explicit invalidation needed
   Risk: stale data for up to TTL seconds
   Best for: mostly-read data (product catalog, country list)

2. Event-based invalidation:
   On write → delete/update cache entry
   Best for: user profiles, session data
   Pattern: Write-through (write to cache AND DB simultaneously)

3. Cache-aside (Lazy Loading):
   Read cache → miss → read DB → write cache → return
   Never stale for long — populated on demand
   Risk: cache stampede on cold start

4. Write-behind (Write-back):
   Write to cache → async flush to DB
   Risk: data loss if cache node fails before flush
   Best for: write-heavy, can tolerate slight lag

Cache Key Patterns

"user:profile:{userId}"
"product:detail:{productId}:{locale}"
"rate:limit:{userId}:{endpoint}:{windowStart}"
"session:token:{sessionId}"

// Version prefix for cache busting during deployment
"v2:user:profile:{userId}"

🎯 CLASSIC SYSTEM DESIGN CHEATSHEETS

Design a Rate Limiter

Problem: Limit each user to N requests per window

Algorithm choices:
  Fixed window:    Simple; burst allowed at window boundary
  Sliding window:  Accurate; uses more memory
  Token bucket:    Smooth rate; allows short bursts
  Leaky bucket:    Strict rate; no bursts

Production implementation (Sliding Window Log in Redis):
  Key:   "ratelimit:{userId}:{endpoint}"
  Type:  Redis Sorted Set (score = timestamp, member = requestId)

  Lua script (atomic):
    1. ZREMRANGEBYSCORE key 0 (now - windowMs)   # remove old entries
    2. count = ZCARD key                          # count in window
    3. if count >= limit → REJECT (429)
    4. ZADD key now requestId                     # record this request
    5. EXPIRE key windowMs                        # auto-cleanup

Storage per user: O(requests in window) — bound by rate limit itself

Distributed: All instances talk to same Redis → consistent rate limiting

Design a Notification System

Requirements: Send 10M push/email/SMS notifications/day

Components:
  Event Sources → Kafka "notification-events"
       ↓
  Notification Service:
    - Loads user preferences (push? email? SMS?)
    - Loads device tokens from Device Token Service
    - Publishes to channel-specific topics
       ↓

Design a Chat System

Requirements: 1-1 and group chat, message history, online status

Real-time delivery: WebSocket (bidirectional, persistent connection)
Fallback:           Long polling for clients that can't WebSocket
Message history:    Cassandra (write-heavy, time-ordered, horizontally scalable)

Message Schema:
  message_id   TIMEUUID    -- time-sortable unique ID
  sender_id    UUID
  chat_id      UUID        -- conversation ID
  content      TEXT
  created_at   TIMESTAMP

Delivery flow:
  1. Sender → WebSocket Server A → writes to Message DB
  2. Server A → Kafka "messages" topic
  3. Kafka → Message Fanout Service

Design a URL Shortener

Requirements: Create short links, redirect, track clicks

ID generation:
  Option A: Base62(auto-increment ID) — sequential, predictable
  Option B: Base62(random 7 chars)    — unpredictable, no ID leakage
  Option C: Consistent hash of long URL — same URL = same short code

Storage: PostgreSQL
  short_code  VARCHAR(8)  PRIMARY KEY
  long_url    TEXT        NOT NULL
  user_id     BIGINT
  created_at  TIMESTAMP
  expires_at  TIMESTAMP   (nullable — permanent links have no expiry)
  click_count BIGINT DEFAULT 0  -- approximate, updated async

Redirect flow (latency critical — must be fast):
  1. GET /abc1234
  2. Redis cache lookup (key: "url:abc1234", TTL: 24h) → HIT → redirect
  3. Cache miss → PostgreSQL lookup → cache → redirect
  4. HTTP 301 (browser caches permanently) or 302 (we track all clicks)

Analytics (async — don't block redirect):
  Redirect → publish click event to Kafka

Design a Payment System

Requirements: Process payments, exactly-once, audit trail, multi-currency

Exactly-once guarantee:
  1. Client sends Idempotency-Key header (UUID)
  2. Payment Service checks Redis: key exists? → return cached result
  3. Key not found → process payment (single time)
  4. Store result in Redis with TTL 24h
  5. Write to payment_records with UNIQUE(idempotency_key)
     → DB constraint as final safety net

Transaction states (state machine):
  PENDING → PROCESSING → COMPLETED
                      ↘ FAILED
                      ↘ REFUNDED

Outbox pattern (atomic event publishing):
  BEGIN TRANSACTION;
    INSERT INTO payments (id, amount, status) VALUES (...);
    INSERT INTO outbox_events (payload, topic) VALUES (..., 'payment-processed');
  COMMIT;
  -- Separate process publishes outbox → Kafka

Currency handling:
  NEVER use float/double for money
  Store as BIGINT (smallest currency unit: cents)
  Display: amount / 100 = $10.00 (divide at presentation layer only)

Reconciliation:
  Daily job: compare payment_records with bank statement
  Discrepancies → alert finance team → manual investigation
  Reconciliation window: T+1 (next business day)

Fraud detection:
  Velocity checks: N transactions per hour per user → flag
  Geo-velocity: card used in AU then SG 5 minutes later → flag

Design a Feed/Timeline System

Requirements: Twitter/Facebook-style feed — show posts from followed users

Two approaches:

PUSH (fan-out on write):
  User posts → write to ALL followers' feeds immediately
  Read: O(1) — just read your pre-computed feed
  Write: O(N) — N followers × write
  Problem: celebrities with 10M followers = 10M writes per post

PULL (fan-out on read):
  User posts → write to own timeline only
  Read: merge N followed users' timelines → sort by time
  Read: O(N) — N followees × read per page load
  Problem: following 1000 users = 1000 reads per page load

HYBRID (what Twitter/Instagram actually do):
  Normal users: PUSH to followers' feeds
  Celebrities (>10k followers): PULL at read time, merged client-side
  Celebrities' posts cached separately — one write, read from cache

Storage:
  Feed items: Redis Sorted Set (score=timestamp, member=postId)
  "feed:{userId}" → ZRANGE -20 0 → 20 most recent post IDs
  Post details: Separate cache/DB lookup (IDs only in feed)

Pagination:
  Use cursor (last seen postId), not offset
  Inserting new posts doesn't shift pages

🔑 CONSISTENCY PATTERNS CHEATSHEET

STRONG CONSISTENCY:
  Every read reflects the most recent write
  Implementation: Single master + synchronous replication
  Cost: Higher latency (must wait for replication)
  Use for: Financial transactions, inventory (can't oversell)

EVENTUAL CONSISTENCY:
  After some time, all replicas converge to same value
  Implementation: Async replication, CRDT, gossip protocol
  Cost: Reads may be stale
  Use for: Social feeds, product views, analytics counters

READ-YOUR-WRITES CONSISTENCY:
  After a write, the same user always reads the latest value
  Implementation: Route user's reads to the same replica they wrote to
  Or: Use a version token — only read from replica that has this version
  Use for: Profile updates (user should see their own change immediately)

CAUSAL CONSISTENCY:
  Events that are causally related appear in the correct order
  Example: Comment appears after the post it's replying to
  Implementation: Vector clocks, logical timestamps
  Use for: Comment threads, collaborative editing

MONOTONIC READ CONSISTENCY:
  Once a user has seen version X, they never see an older version
  Implementation: Sticky sessions to same replica

LOAD BALANCER:
  L4 (TCP): Low latency, no HTTP awareness — use for non-HTTP
  L7 (HTTP): URL routing, header inspection, SSL termination — use for APIs
  AWS ALB: L7, HTTP/2, WebSocket, sticky sessions
  AWS NLB: L4, ultra-low latency, static IP

MESSAGE BROKER:
  SQS: Managed, simple queue, at-least-once
  Kafka: Event streaming, replay, high throughput, ordering
  RabbitMQ: Complex routing, pub-sub, RPC patterns
  Pub/Sub: GCP-native, globally scalable

CACHE:
  Redis: Rich data structures, Lua scripting, pub-sub
  Memcached: Simple K-V, multi-threaded, pure caching
  CDN (CloudFront/Cloud CDN): Static assets, geographic distribution

OBJECT STORAGE:
  S3/GCS: Unlimited scale, 99.999999999% durability, cheap
  Use for: Files, backups, media, static assets, Terraform state

SEARCH:
  Elasticsearch: Full-text, faceted, geo — industry standard
  OpenSearch: AWS-managed Elasticsearch fork
  Algolia: Managed, developer-friendly, expensive at scale

🚨 FAILURE MODES CHEATSHEET

CASCADE FAILURE:
  One slow service → threads blocked → upstream queues grow → entire system down
  Prevention: Circuit breaker + bulkhead + timeout on every external call

THUNDERING HERD:
  Many clients retry simultaneously → overwhelm recovering service
  Prevention: Exponential backoff + jitter on retries
  Formula: sleep = min(cap, base * 2^attempt) + random(0, jitter)

SPLIT BRAIN:
  Network partition → two nodes think they are the primary
  Prevention: Quorum-based consensus (etcd, Zookeeper)
  Only proceed if majority of nodes agree

CACHE STAMPEDE:
  Popular cache key expires → many requests hit DB simultaneously
  Prevention: Mutex lock (only one request rebuilds), TTL jitter, early refresh

MEMORY LEAK:
  ThreadLocal not cleared, listener not unregistered, static collection grows
  Detection: JVM heap trending up over time in CloudWatch/Prometheus
  Prevention: Heap dump analysis with MAT or VisualVM

HOT PARTITION (Kafka/DynamoDB):
  All traffic routed to one partition/shard due to poor key choice
  Example: Partitioning by date → all today's writes hit same partition
  Fix: Add high-cardinality field to partition key (userId + date)
